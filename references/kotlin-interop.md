# JSpecify + Kotlin Interop

## How Kotlin Sees JSpecify Annotations

The Kotlin compiler (K2, 2.0+) has built-in support for JSpecify annotations. When Java code is annotated with JSpecify:

- `@Nullable` → Kotlin sees the type as nullable (`String?`)
- Inside `@NullMarked` scope, unannotated Java types → Kotlin sees them as non-null (`String`, not `String!` platform type)
- Outside `@NullMarked` scope (or `@NullUnmarked`), unannotated Java types → Kotlin sees them as platform types (`String!`)

This eliminates the "platform type" ambiguity that causes hidden NPEs when calling Java from Kotlin.

## Kotlin Compiler Configuration

### Gradle (Kotlin DSL)

Enable strict JSpecify mode in `build.gradle.kts`:

```kotlin
kotlin {
  compilerOptions {
    freeCompilerArgs.add("-Xjspecify-annotations=strict")
    // Also enable null checks for non-strict annotations if desired:
    // freeCompilerArgs.add("-Xtype-enhancement-improvements-strict-mode")
  }
}
```

The `kotlinOptions` DSL is deprecated since Kotlin 1.8 and removed in K2. Use `compilerOptions` above.

### Maven (Kotlin Maven Plugin)

```xml
<plugin>
  <groupId>org.jetbrains.kotlin</groupId>
  <artifactId>kotlin-maven-plugin</artifactId>
  <configuration>
    <args>
      <arg>-Xjspecify-annotations=strict</arg>
    </args>
  </configuration>
</plugin>
```

## Annotating Kotlin Code for Java Consumers

Kotlin's type system already distinguishes nullable (`String?`) and non-null (`String`). When Kotlin code is consumed from Java, JSpecify annotations are not automatically emitted.

To add JSpecify annotations to Kotlin declarations for Java callers, annotate explicitly:

```kotlin
import org.jspecify.annotations.Nullable
import org.jspecify.annotations.NullMarked

// Annotate at file level
@file:NullMarked

fun findUser(id: String): User = ...

// Or per-function for mixed files
fun findByName(name: String): @Nullable User? = ...
```

**Important:** `@file:NullMarked` affects the **compiled class-file metadata** — it tells Java tooling (NullAway, IDEs, other JVM consumers) that declarations in this file are non-null by default. It does _not_ change Kotlin's own type system or enable additional Kotlin null-safety checks; the Kotlin compiler already enforces nullability through `?` suffixes. Use `@file:NullMarked` when your Kotlin code is consumed from Java and you want Java tooling to see accurate nullability contracts.

## Mixed Java/Kotlin Projects

### Recommended setup

1. Annotate Java packages with `@NullMarked` as described in the main workflow
2. Enable `-Xjspecify-annotations=strict` in the Kotlin compiler
3. Kotlin code calling into `@NullMarked` Java APIs gets accurate nullability — no platform types
4. For Kotlin-to-Java calls, rely on Kotlin's own type system (`?` suffix) — Java NullAway will respect Kotlin's `@Nullable`/`@NonNull` on generated bytecode

### KSP / kapt interaction

If using KAPT or KSP for annotation processing, JSpecify annotations are passed through and visible to annotation processors. NullAway running on the Java compilation step still analyzes the Java sources independently.

### Spring Boot projects (Kotlin)

Spring Boot 4.x (Spring Framework 7.x) marks its own APIs with `@NullMarked`. Kotlin code calling Spring Boot 4.x APIs will see non-null types by default once `-Xjspecify-annotations=strict` is enabled. Spring Boot 3.x uses Spring's own `org.springframework.lang` annotations, which are not JSpecify — those APIs will still appear as platform types.

```kotlin
// Before: result is String! (platform type)
val result = environment.getProperty("key")

// After (with strict mode): result is String? (properly nullable)
val result: String? = environment.getProperty("key")
// Kotlin compiler now warns if you use result without null check
```

## Gotchas

- **K1 compiler (pre-2.0)**: K1 reached end-of-life with Kotlin 2.0 (2024). If you are pinned to Kotlin < 2.0, JSpecify support is experimental — expect gaps.
- **`@NullUnmarked` packages**: Kotlin still sees platform types for Java code in `@NullUnmarked` scope — same behavior as unannotated code.
- **Collections**: `List<@Nullable String>` in Java maps to `List<String?>` in Kotlin with strict mode. Without strict mode, it maps to `List<String!>`.
- **Arrays**: `@Nullable String[]` → `Array<String?>?` in Kotlin. The array itself can also be null if the parameter/field is `@Nullable`.
